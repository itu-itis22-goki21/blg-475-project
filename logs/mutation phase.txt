        int b = 0;
        int c = 2;
        int d = 0;
        int e = 2;
        for (int i = 5; i <= n; i++) {
            int next = a + b + c + d;
            a = b;
            b = c;
            c = d;
            d = e;
            e = next;
        }
        return e;

    private static int fib4Expected(int n) {
        if (n == 0 || n == 1) {
            return 0;
        int[] values = {0, 0, 2, 0, 2};
        if (n < values.length) {
            return values[n];
        }
        if (n == 2) {
            return 2;
        }
        if (n == 3) {
            return 0;
        }
        if (n == 4) {
            return 2;
        }
        int a = 0;
        int b = 0;
        int c = 2;
        int d = 0;
        int e = 2;
        int[] dp = new int[n + 1];
        System.arraycopy(values, 0, dp, 0, values.length);
        for (int i = 5; i <= n; i++) {
            int next = a + b + c + d;
            a = b;
            b = c;
            c = d;
            d = e;
            e = next;
            dp[i] = dp[i - 1] + dp[i - 2] + dp[i - 3] + dp[i - 4];
        }
        return e;
        return dp[n];
    }

Why these mutations improve effectiveness
These mutations improve the tests because they now exercise the places where faulty implementations usually survive base examples:

Boundary sensitivity: off-by-one mistakes and “happy path only” logic are much more likely to be caught now. task44, task46, task48, task55, and task124 are especially stronger here.
Validation behavior: many files now check that invalid domains fail instead of silently producing a value. That makes it harder for permissive but wrong implementations to pass.
Behavioral distinctions inside the same valid domain: the new cases separate tied vs unique matches, adjacent vs separated groups, duplicates vs no duplicates, sorted vs unsorted intersections, and precedence/tie-breaking rules. Those are exactly the mutations that reveal wrong branch logic without needing to inspect the production code.
Better failure localization: replacing aggregated boolean lists with direct assertions means a failure points to a specific partition, not just “some case in this file failed.”
One constraint I respected: for workbook rows like “non-int” or “non-string” on APIs whose Java signatures already enforce the type statically, I did not add reflection-based tests. Instead I covered the closest meaningful invalid runtime partitions, such as null, malformed structure, illegal numeric ranges, or mismatched collection shapes.
